# 📦 Caixa de Pandora da Estatística e Ciência de Dados da UFPR

Bem-vindo(a) à **Caixa de Pandora da Estatística e Ciência de Dados da UFPR**.

Este repositório existe para reunir, organizar e publicar as datas de provas, segundas chamadas, trabalhos, entregas, seminários, apresentações e outras atividades avaliativas do curso de Estatística e Ciência de Dados da UFPR.


## 🎯 Objetivo do projeto

O curso tem muitas disciplinas, muitas provas, muitas entregas e, como toda boa experiência estatística, uma quantidade não desprezível de incerteza.

Este projeto busca reduzir a entropia acadêmica criando um calendário colaborativo, versionado e publicado em formato acessível.

Em termos práticos, queremos manter um arquivo Markdown por semestre, por exemplo:

- `calendario/2026-1.md`
- `calendario/2026-2.md`
- `calendario/2027-1.md`

Esses arquivos são convertidos e publicados como uma página HTML via GitHub Pages.

## 🧪 O que pode ser registrado aqui?

Podem ser registradas informações sobre:

- provas;
- segundas chamadas;
- trabalhos;
- listas de exercícios;
- apresentações;
- seminários;
- relatórios;
- entregas parciais;
- entregas finais;
- mudanças de data;
- mudanças de sala;
- publicação de notas ou resultados;
- links oficiais para gabaritos, notas ou comunicados.

## 📌 Arquivo principal do semestre atual

O arquivo principal do semestre 2026/1 é:

- `calendario/2026-1.md`

Sempre que possível, concentre as contribuições do semestre atual nesse arquivo.

## 🧭 Como contribuir

Há três formas principais de contribuir.

Você pode escolher a forma que for menos dolorosa para o seu nível atual de relacionamento com o GitHub.

---

# 1. 🟢 Contribuir abrindo uma issue

Esta é a forma mais simples.

Você não precisa editar arquivo, entender Markdown, fazer fork, criar branch, fazer commit, nem conversar com espíritos ancestrais do Git.

Use esta opção quando você apenas quer informar uma data, corrigir uma informação ou avisar que algo mudou.

## Passo a passo

1. Acesse a página principal deste repositório no GitHub.

2. Clique na aba **Issues**.

3. Clique em **New issue**.

4. No título da issue, escreva algo objetivo.

   Exemplos:

   - `Adicionar prova de Pré-Cálculo`
   - `Corrigir sala da prova de Estatística Básica`
   - `Adicionar entrega do trabalho de Fundamentos de Programação`
   - `Atualizar segunda chamada de Estatística e Sociedade`

5. No corpo da issue, informe o máximo possível dos dados abaixo:

   - **Disciplina:**
   - **Semestre da disciplina:**
   - **Tipo de avaliação:** prova, trabalho, segunda chamada, apresentação, seminário etc.
   - **Nome da avaliação:** Prova 1, Prova 2, Trabalho 1 etc.
   - **Data:**
   - **Horário:**
   - **Sala ou local:**
   - **Conteúdo:**
   - **Comentários adicionais:**
   - **Fonte da informação:** Moodle, professor(a), plano de ensino, e-mail, mensagem oficial etc.
   - **Link para notas/resultados, se houver:**

6. Clique em **Submit new issue**.

7. Um mantenedor revisará a informação e adicionará ao arquivo correspondente.

## Exemplo de issue

Título:

- `Adicionar Prova 1 de Pré-Cálculo`

Corpo:

- **Disciplina:** Pré-Cálculo
- **Semestre:** 1º semestre
- **Tipo de avaliação:** Prova
- **Nome da avaliação:** Prova 1
- **Data:** 2026-04-15
- **Horário:** 19:00
- **Sala:** PA-08
- **Conteúdo:** funções, equações, inequações e trigonometria básica
- **Comentários:** prova sem consulta
- **Fonte:** comunicado do professor no Moodle
- **Notas/resultados:** a definir

# 2. 🟡 Contribuir editando diretamente o Markdown pelo GitHub

Esta é a melhor opção para quem quer corrigir diretamente o calendário, mas ainda prefere usar apenas a interface web do GitHub.

Você não precisa instalar Git no computador.

Você não precisa usar terminal.

Você não precisa clonar o repositório.

Você precisa apenas editar o arquivo no navegador.

## Quando usar esta opção?

Use esta opção quando você quer:

- adicionar uma avaliação diretamente no arquivo;
- corrigir uma data;
- corrigir uma sala;
- corrigir o conteúdo de uma prova;
- adicionar um link para notas ou resultados;
- completar informações que estavam como `a definir`.
- etc

## Arquivo a editar

Para o semestre 2026/1, edite:

- `calendario/2026-1.md`

Link do arquivo no repositório:

https://github.com/SEU-USUARIO-OU-ORGANIZACAO/NOME-DO-REPOSITORIO/blob/main/calendario/2026-1.md

Substitua `SEU-USUARIO-OU-ORGANIZACAO` e `NOME-DO-REPOSITORIO` pelos dados reais deste repositório.

## Passo a passo

1. Acesse o arquivo `calendario/2026-1.md`.

2. Clique no ícone de lápis, geralmente chamado **Edit this file**.

3. Faça a alteração necessária.

4. Mantenha a estrutura do arquivo.

   Exemplo de estrutura esperada:

   - semestre;
   - disciplina;
   - prova ou trabalho;
   - item específico;
   - data;
   - horário;
   - sala;
   - conteúdo;
   - comentários;
   - notas/resultados.

5. Role até o final da página.

6. Escreva uma mensagem curta de commit.

   Exemplos:

   - `Adiciona prova de Pré-Cálculo`
   - `Corrige sala de Estatística Básica`
   - `Adiciona trabalho de Fundamentos de Programação`
   - `Atualiza segunda chamada de Estatística e Sociedade`

7. Escolha a opção para criar uma nova branch e abrir um pull request, quando o GitHub oferecer essa opção.

8. Clique em **Propose changes**.

9. Depois, clique em **Create pull request**.

10. Aguarde a revisão de um mantenedor.

## Boas práticas para editar o Markdown

- Não apague informações de outras disciplinas sem justificativa.
- Não substitua uma data confirmada por uma informação incerta sem explicar a fonte.
- Use `a definir` quando a informação ainda não estiver confirmada.
- Use datas no formato `AAAA-MM-DD`, por exemplo `2026-04-15`.
- Use horários no formato `HH:MM`, por exemplo `19:00`.
- Quando houver dúvida, registre a dúvida em **Comentários**.
- Quando uma data mudar, registre também no histórico de alterações importantes do arquivo.

# 3. 🔵 Contribuir fazendo fork, edição local e pull request

Esta é a opção mais completa.

Use esta opção se você já está confortável com Git, GitHub, branches e pull requests, ou se quer treinar essas habilidades em um projeto real, útil e relativamente seguro.

Nada como aprender Git tendo como motivação não perder a data da prova.

## Quando usar esta opção?

Use esta opção quando você quer:

- fazer várias alterações de uma vez;
- revisar o Markdown localmente;
- editar com VS Code, Vim, Emacs, RStudio, Zed ou outro editor;
- rodar scripts de validação localmente;
- contribuir com melhorias estruturais no projeto;
- alterar documentação, workflows, templates ou scripts.

## Passo a passo

1. Acesse a página principal deste repositório no GitHub.

2. Clique em **Fork**.

3. Crie uma cópia do repositório na sua conta.

4. Clone o seu fork para o computador:

   `git clone https://github.com/SEU-USUARIO/NOME-DO-REPOSITORIO.git`

5. Entre no diretório do projeto:

   `cd NOME-DO-REPOSITORIO`

6. Crie uma branch para sua alteração:

   `git checkout -b adiciona-datas-2026-1`

7. Edite o arquivo desejado.

   Para o semestre 2026/1:

   `calendario/2026-1.md`

8. Revise suas alterações:

   `git diff`

9. Adicione os arquivos modificados:

   `git add calendario/2026-1.md`

10. Faça o commit:

   `git commit -m "Adiciona datas de avaliações de 2026-1"`

11. Envie sua branch para o seu fork:

   `git push origin adiciona-datas-2026-1`

12. Acesse o GitHub.

13. Abra um pull request do seu fork para o repositório original.

14. Descreva claramente o que foi alterado.

15. Aguarde revisão.

# 🧱 Estrutura dos arquivos

A estrutura planejada do projeto é:

- `README.md`: explicação geral do projeto.
- `calendario/`: arquivos Markdown dos semestres.
- `calendario/2026-1.md`: calendário do semestre 2026/1.
- `.github/ISSUE_TEMPLATE/`: modelos para abertura de issues.
- `.github/workflows/`: automações do GitHub Actions.
- `docs/`: arquivos publicados no GitHub Pages, se o projeto usar publicação por pasta.
- `scripts/`: scripts auxiliares de validação ou conversão, se forem adicionados.

---

# ✍️ Padrão de preenchimento das avaliações

Cada avaliação deve seguir, sempre que possível, o seguinte padrão:

- **Status:** ✅ Confirmado, 🟡 A confirmar, 🔄 Alterado ou ❌ Cancelado
- **Chamada:** 1ª chamada, 2ª chamada ou não aplicável
- **Data:** AAAA-MM-DD
- **Horário:** HH:MM
- **Sala:** sala, laboratório, Moodle, GitHub, e-mail ou outro local
- **Conteúdo:**
  - tópico 1
  - tópico 2
  - tópico 3
- **Comentários:**
  - informações adicionais relevantes
- **Notas/resultados:**
  - link para resultado, gabarito, planilha ou publicação oficial

---

# ✅ Status recomendados

Use os seguintes status:

- ✅ **Confirmado:** informação confirmada por fonte oficial ou comunicação clara do professor.
- 🟡 **A confirmar:** informação ainda incerta, incompleta ou informal.
- 🔄 **Alterado:** informação mudou em relação ao registro anterior.
- ❌ **Cancelado:** avaliação cancelada.

---

# 🧾 Fontes recomendadas

Sempre que possível, informe a fonte da informação.

Fontes úteis:

- Moodle;
- plano de ensino;
- professor(a);
- e-mail institucional;
- mensagem oficial em grupo da disciplina;
- SIGA ou sistema acadêmico equivalente;
- edital ou comunicado da coordenação.

Evite registrar boatos sem marcar claramente como informação a confirmar.

A Caixa de Pandora já é complicada o suficiente sem inferência bayesiana em fofoca acadêmica.

---

# 🧨 O que não colocar neste repositório

Não publique:

- dados pessoais de estudantes;
- justificativas individuais de ausência;
- atestados;
- prints com informações privadas;
- notas individuais de estudantes;
- conversas privadas sem autorização;
- qualquer informação sensível.

Links para resultados oficiais podem ser adicionados quando forem públicos ou acessíveis pelos meios institucionais adequados.

---

# 🔄 Revisão e aprovação

As alterações podem ser revisadas por mantenedores antes de entrar no calendário principal.

O objetivo da revisão não é burocratizar o processo.

O objetivo é evitar:

- data errada;
- sala errada;
- prova duplicada;
- conteúdo mal atribuído;
- alteração sem fonte;
- caos estatisticamente significativo.

# 🧑‍🔧 Mantenedores

Os mantenedores são responsáveis por:

- revisar pull requests;
- verificar se as informações estão minimamente completas;
- organizar o arquivo do semestre;
- fechar issues já resolvidas;
- corrigir problemas de formatação;
- manter o site publicado funcionando;
- impedir que a Caixa de Pandora vire um experimento não controlado.

---

# 🧠 Filosofia do projeto

Este projeto segue alguns princípios simples:

1. Informação acadêmica importante deve ser fácil de encontrar.
2. Contribuir deve ser mais fácil do que reclamar no grupo.
3. O histórico das alterações deve ser preservado.
4. A fonte da informação deve ser registrada sempre que possível.
5. A organização coletiva reduz ansiedade individual.
6. Calendário bom é calendário que alguém consegue usar na semana da prova.
